# Does Hashing Make Text Look Random?

This post is the first part of a small experiment series on statistical structure in text, hashes, and encrypted data. The question is simple: if I take a list of English words and hash each word, does the output start to look random at the byte level?

The answer is mostly yes, but with an important warning. Byte-level KL divergence can show that visible frequency structure disappeared. It cannot prove that a hash function is secure.

Repository / source / reference:

[`https://github.com/vishwaksen-1/words-kld-hash`](https://github.com/vishwaksen-1/words-kld-hash)

---

## What this post is about

This post compares plaintext word data against several hash and checksum outputs using:

- byte-frequency distributions
- Shannon entropy
- KL divergence to a uniform byte distribution

It does not attempt to test collision resistance, preimage resistance, avalanche behavior, or cryptographic strength. 
The goal is narrower: measure how much obvious byte-frequency structure remains.

---

## Background

Plain English text is not byte-random. In this dataset, most bytes are lowercase ASCII letters, with a few separators. That means the byte distribution is concentrated in a small part of the `0..255` range.

KL divergence gives a way to compare one distribution with another. Here I use it mostly in one direction:

```text
KL(method || uniform)
```

If the value is high, the output distribution is far from uniform. If it is close to zero, the output is close to uniform at this specific measurement level.

That last phrase matters: "at this measurement level." A byte-frequency test is only one lens.

---

## Why this matters

Hash functions are supposed to destroy visible structure in their inputs. If similar-looking English words produced byte distributions that still looked like English, that would be a very bad sign.

This experiment is useful because it separates two ideas that are easy to blur together:

- statistical uniformity in a simple experiment
- actual cryptographic security

Those are related, but they are not the same thing. MD5 and SHA-1 can look excellent by byte KL while still being broken for collision resistance.

---

## System / Approach / Design

The input is a wordlist containing `7776` words taken from the EFF Diceware wordlist. The plaintext version is created by joining the words into bytes, producing `54368` plaintext bytes.

For the hash study, each word is hashed independently. I compared:

- DJB2: 32-bit non-cryptographic hash
- CRC-24: 24-bit checksum
- MD5: 128-bit broken cryptographic hash
- SHA-1: 160-bit broken cryptographic hash
- SHA-256: 256-bit SHA-2 hash
- SHA-512: 512-bit SHA-2 hash

Important decisions:

- Decision 1: compare outputs to uniform bytes.
- Why: a strong hash output should not preserve English byte frequencies.
- Alternative: compare only hash outputs against plaintext.
- Tradeoff: uniform comparison is easier to read, but it is not a security proof.

- Decision 2: include broken hashes.
- Why: MD5 and SHA-1 show the difference between "looks random here" and "is cryptographically safe."
- Alternative: include only SHA-2.
- Tradeoff: the post needs a stronger caveat, but the lesson is better.

## Implementation / What I did

The main experiment is generated by [`h1-test2.py`](https://github.com/vishwaksen-1/words-kld-hash/blob/master/h1-test2.py). The hash family is defined like this:

```python
HASH_ALGORITHMS = [
    ("DJB2", "32-bit non-crypto", "djb2"),
    ("CRC-24", "24-bit checksum", "crc24"),
    ("MD5", "128-bit broken crypto", "md5"),
    ("SHA-1", "160-bit broken crypto", "sha1"),
    ("SHA-256", "256-bit", "sha256"),
    ("SHA-512", "512-bit", "sha512"),
]
```

For each output, the script counts byte frequencies, computes entropy, and computes KL divergence to a uniform distribution over all 256 byte values.

## Tests / Observations

| Method | Digest bytes/word | Entropy bits/byte | KL(method \|\| uniform) |
| --- | ---: | ---: | ---: |
| Plain | - | 4.2691 | 2.5860 |
| DJB2 | 4 | 7.9111 | 0.0616 |
| CRC-24 | 3 | 7.9919 | 0.0056 |
| MD5 | 16 | 7.9985 | 0.0011 |
| SHA-1 | 20 | 7.9987 | 0.0009 |
| SHA-256 | 32 | 7.9993 | 0.0005 |
| SHA-512 | 64 | 7.9996 | 0.0003 |

The plaintext has only `4.2691` bits of entropy per byte and a KL-to-uniform value of `2.5860`. That is the expected signature of English-like byte data.

The cryptographic hashes are much closer to uniform. SHA-256 lands at `7.9993` bits of entropy per byte with `KL(method || uniform) = 0.0005`.

DJB2 improves dramatically over plaintext, but it is still much farther from uniform than the cryptographic hash outputs in this experiment.

---

## What failed / What was surprising

What broke:

- A simple byte-frequency result can make weak or broken algorithms look better than they are.

Why it happened:

- KL divergence over byte counts does not know anything about collisions, preimages, length extension, chosen-prefix attacks, or the internal design of a hash function.

How I diagnosed it:

- MD5 and SHA-1 looked close to uniform even though both are known to be cryptographically broken for collision resistance.

What changed:

- I kept the metric, but narrowed the claim. The result is about visible byte-frequency structure, not security.

What tradeoff I accepted:

- The experiment remains simple and readable, but the interpretation has to be disciplined.

---

## Result

- What works: hashing destroys the visible byte-frequency structure of the wordlist.
- What partially works: KL-to-uniform gives a useful first-pass statistical baseline.
- What does not work yet: this does not rank cryptographic security.
- What is untested: collision behavior, avalanche behavior, preimage resistance, and structured adversarial inputs.

The main result is that SHA-256 and SHA-512 look almost uniform at the byte level, while plaintext is visibly structured.

---

## What I learned

I expected plaintext to be far from uniform, and the measurement confirmed it clearly.

I also expected cryptographic hashes to be close to uniform. The more useful lesson was seeing MD5 and SHA-1 pass this visual/statistical check while still being unsafe for modern cryptographic use.

We obviously see that hashes flatten byte frequencies. But the main lesson is that a metric can be true, useful, and still incomplete.
